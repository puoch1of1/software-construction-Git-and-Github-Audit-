In this example, a broken Python script (disk_usage.py) is used to demonstrate how diff and patch work in practice. Before fixing the bugs, two copies of the script are created: one kept unchanged for comparison and another used for making fixes. Running the script first helps identify errors instead of guessing what’s wrong.

The first issue is a syntax error caused by a return statement placed outside a function, which is invalid in Python. This is fixed by replacing the return with sys.exit, allowing the script to exit properly with a status code. After fixing that, the script still behaves incorrectly by reporting low disk space even when sufficient space is available.

A closer look at the code reveals a logic error: disk space is being converted to gigabytes twice—once when calling the function and again inside the function itself. Removing one of these conversions fixes the logic bug and allows the script to work as intended.

Once the script is fixed, a diff file is generated to capture only the changes made. This diff file can then be shared with a colleague, who can apply the fixes to their own copy of the script using the patch command. After applying the patch, the script runs successfully.

This example shows how diff and patch can be used to identify changes, package fixes, and apply them elsewhere. However, it also highlights how manual this process is, setting the stage for why version control systems like Git are more powerful and efficient for managing changes and collaboration.

### **Summary and Notes: Using `diff` and `patch` to Fix a Script**

---

## **1. Problem Overview**

A colleague requests help fixing a Python script named `disk_usage.py`. The purpose of the script is to check disk space usage and display an error if there is insufficient space for normal operation. However, the script contains several bugs that prevent it from working correctly.

This scenario is used to demonstrate how to:

* Identify bugs
* Fix code safely
* Share fixes using `diff`
* Apply fixes using `patch`

---

## **2. Preparing to Fix the Script**

Before making any changes:

* Two copies of the script are created:

  * `disk_usage_original.py`: kept unchanged for reference and comparison
  * `disk_usage_fixed.py`: used to apply and test fixes

This approach ensures the original file remains intact and allows easy comparison of changes.

---

## **3. Identifying the First Bug**

When the script is executed, Python raises a syntax error:

* **Error:** `return outside of function`

### **Cause**

In Python, `return` statements are only valid inside functions. The script contains a `return` statement outside of any function.

### **Possible Fixes**

1. Wrap the existing code inside a function and call it from the main script.
2. Replace the `return` statement with `sys.exit()`, which exits the program with a status code.

### **Chosen Solution**

* The `return` statement is replaced with `sys.exit()` to properly exit the script.

After this change, the syntax error is resolved.

---

## **4. Identifying the Second Bug**

After fixing the syntax error, the script runs but incorrectly reports insufficient disk space.

### **Cause**

* The script converts disk space from bytes to gigabytes twice.
* The function `check_disk_usage()` already converts bytes to gigabytes internally.
* The function call incorrectly passes a value already converted to gigabytes using `2 * 2**30`.

### **Explanation**

* `2**30` represents the number of bytes in one gigabyte.
* Passing `2 * 2**30` assumes the function expects bytes.
* Since the function already performs this conversion, the disk space threshold is reduced incorrectly.

### **Fix**

* Remove the unnecessary conversion when calling the function.
* Pass the threshold value directly in gigabytes.

After this correction, the script works as expected.

---

## **5. Sharing the Fix Using `diff`**

To send the fix to the colleague:

* A diff file is generated by comparing:

  * `disk_usage_original.py`
  * `disk_usage_fixed.py`

The diff file contains only the changes made to fix the bugs.

The contents of the diff file can be reviewed using the `cat` command.

---

## **6. Applying the Fix Using `patch`**

The colleague can apply the fix by:

* Running the `patch` command with the diff file.
* This updates their original script automatically with the correct changes.

After applying the patch, the script executes successfully.

---

## **7. Key Concepts Demonstrated**

* Running code first to observe errors before modifying it
* Keeping an unchanged original file for comparison
* Using `diff` to capture code changes
* Using `patch` to apply fixes cleanly
* Understanding how small logic errors can cause major issues

---

## **8. Important Takeaway**

While `diff` and `patch` are powerful tools for sharing and applying changes, this process is still manual. Version control systems such as Git automate and improve this workflow, making collaboration safer, faster, and more reliable.
